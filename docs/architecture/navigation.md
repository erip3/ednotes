# Technical Specification: Hybrid Knowledge Graph & Navigation System

## Project Overview

This project transforms a traditional hierarchical CMS into a Hybrid Knowledge Graph. It maintains a strict category-based hierarchy for organization and "folder-style" browsing, while overlaying a Directed Acyclic Graph (DAG) for prerequisite mapping and cross-category associations.

## Database Schema (Supabase / PostgreSQL)

### The Categories (The Tree)

The hierarchy is maintained via a self-referencing parent_id. The is_topic flag marks a category as a major landing page/subject area.

```
create table public.categories (
  id bigint generated by default as identity not null,
  title text not null,
  parent_id bigint null,
  published boolean default true,
  "order" bigint null,
  is_topic boolean not null default false, -- Marks the root of a "Topic Sidebar"
  topic_id bigint null,                    -- References the parent Topic for inheritance
  constraint categories_pkey primary key (id),
  constraint categories_parent_id_fkey foreign key (parent_id) references categories (id) on update cascade on delete set null
);
```

### The Articles (The Nodes)

Articles belong to a primary_category_id to ensure they always have a "Home" for breadcrumbs. We use jsonb for the content to support structured data and efficient querying.

```
create table public.articles (
  id bigint generated by default as identity not null,
  title text not null,
  content jsonb not null, 
  primary_category_id bigint references categories(id),
  published boolean default false,
  "order" bigint null,
  created_at timestamp with time zone default now(),
  constraint articles_pkey primary key (id)
);
```

### The Connections (The Edges)

This table defines the "Knowledge Graph" dependencies.

```
create table public.article_connections (
  source_id bigint not null references public.articles(id) on delete cascade,
  target_id bigint not null references public.articles(id) on delete cascade,
  connection_type text default 'background', -- 'prerequisite', 'related', 'math_basis'
  constraint article_connections_pkey primary key (source_id, target_id)
);
```

## Navigation Systems

### The "Topic" Sidebar (Contextual Tree + Graph)

When a user views an article within a Topic flow, a sidebar renders the hierarchical structure of that topic, but injects "Graph Edges" (Prerequisites) directly into the tree.

**In-line Prerequisite Injection:** Prerequisites are nested under the article that requires them in the sidebar menu.

**Sticky Context Logic:** To prevent the sidebar from disappearing when navigating to a prerequisite in a different category:

1. The UI checks for a topic_context URL parameter.

2. If present, the sidebar renders the tree for that topic_id, regardless of the current article's actual category.

**Visual Distinction:** Prerequisites from outside the current topic are visually flagged (e.g., an icon or specific color).

### The Global "Return" Arrow

The navigation bar features a primary action button (Arrow) that changes behavior based on the user's path.

**Standard Context:** If browsing normally, the Arrow navigates to the parent_id of the current category.

**Graph Context:** If the URL contains a from_article parameter, the Arrow acts as a return Portal.
It navigates the user directly back to the article that triggered the prerequisite deep-dive.

## Technical Implementation Notes

### State Management (React)

We use URL Search Parameters as the global state for navigation to ensure persistence across refreshes.

```
const [searchParams] = useSearchParams();
const activeTopicId = searchParams.get('topic_context') || currentArticle.topicId;
const fromArticleId = searchParams.get('from_article');
```

### Backend Strategy (Spring Boot)

The API must provide a specialized DTO for the sidebar and article views.

**Recursive CTE:** Used in the database to fetch the full category tree for a Topic.

**Graph Fetch:** When loading a Topic sidebar, the API returns the tree AND a map of article_connections where source_id is within the tree.

### Security & Persistence (Supabase)

**RLS (Row Level Security):** Policies ensure that while the public can read ```published = true``` articles, only the article_editor user can perform INSERT/UPDATE/DELETE.

**Content Storage:** The jsonb field stores the document state, allowing the editor to maintain rich text structures (like TipTap deltas).
